## Overview

Agent007 ships 8 expert agents. Each one declares:

- A **model** tier (opus / sonnet) â€” chosen for the work, not the agent
- A **tool profile** (`minimal` / `coding` / `full`) enforced by `tool-policy-guard` at runtime
- **Trigger keywords** for auto-routing
- **Handoff rules** to other agents and to the human
- **`done_when`** + **`forbidden`** clauses checked by the orchestrator

Agents are dispatched by `/dev`, `/consult`, and `/orchestrate`. They are not called directly by the user.

## Tool profiles (least privilege)

`tool-policy-guard` (PreToolUse/Write|Edit) enforces these at runtime â€” a misrouted agent is blocked before the tool call lands.

| Profile | Tools | Agents |
|---|---|---|
| `minimal` | Read, Glob, Grep | product-expert, security-expert (consulting), code-reviewer (+ Bash for diff stats) |
| `coding` | + Write, Edit, Bash, Agent | backend-db-expert (read-only in v6), frontend-ux-expert (+ WebFetch + WebSearch), platform-expert, refactor-cleaner |
| `full` | everything | loop-operator |

<Warning>
  `backend-db-expert` is intentionally read-only. It analyzes, plans, and consults â€” but a separate agent (or the orchestrator) writes the code. This preserves the architectural review/implementation split.
</Warning>

## Modes

Every agent supports up to 4 output modes. Default depends on the agent:

| Mode | When | Format |
|---|---|---|
| `BUILDER` | Default for `frontend-ux-expert` | Writes code (Write/Edit), runs verifications, never just reviews |
| `PLANNER` | Task breakdown needed | Phased plan with file paths, 2-5 min tasks, verify command per phase |
| `CONSULTANT` | Expert advice ("should I", "compare") | Recommendation + trade-offs + concrete next steps |
| `REVIEWER` | Quality gate | PASS/FAIL per category with file:line evidence; verdict APPROVED / NEEDS FIXES |

## Auto-routing

Agents are auto-selected by keyword match. The agent with the most matches wins. On a tie or with `--experts X,Y`, both respond sequentially with labeled sections.

```
/consult "question"          â†’ keyword match â†’ expert
/dev "task"                  â†’ SDD or trivial â†’ subagent dispatch
/orchestrate <workflow>      â†’ fixed chain per workflow type
```

The full keyword tables live in each agent doc.

## Handoff rules

Each agent declares explicit handoffs. These are not suggestions â€” they are the coordination protocol for `/orchestrate`:

| From | To | When |
|---|---|---|
| backend-db-expert | security-expert | Auth or encryption question |
| backend-db-expert | platform-expert | Deployment or infra question |
| frontend-ux-expert | backend-db-expert | API contract / backend shape |
| frontend-ux-expert | security-expert | Auth or session handling on client |
| code-reviewer | security-expert | OWASP or vulnerability found |
| security-expert | human | CRITICAL vulnerability OR compliance audit (GDPR/SOC2) |
| loop-operator | human | 3 consecutive identical errors OR token budget > 80% |

In `/orchestrate`, every handoff produces a `HANDOFF.md` document at `.sdlc/handoffs/<workflow-id>/<agent>-handoff.md`. The next agent reads it before starting.

## Adding a new agent

Define the agent in `.claude/agents/<name>.md`:

```yaml
---
name: my-agent
description: "One-line elevator pitch."
model: sonnet | opus
tools: [Read, Grep, Glob, ...]
triggers: [keyword1, keyword2, ...]
skills: [skill1, skill2]
handoffs:
  - to: other-agent
    when: "concrete condition"
done_when:
  - "Observable success criterion 1"
forbidden:
  - "Anti-pattern that must never happen"
---

# Agent Name

Identity paragraph...
```

Then register in `.claude/agents/INDEX.md` and add trigger keywords to the routing tables in `CLAUDE.md`.

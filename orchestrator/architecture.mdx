## Overview

Agent007's orchestration runs **entirely inside Claude** — there is no separate TypeScript binary or CLI engine. The orchestrator is CLAUDE.md itself: a set of instructions that Claude follows to classify tasks, route them, dispatch subagents, and enforce quality gates.

The building blocks are:

| Layer | What it is | Where it lives |
|-------|-----------|----------------|
| **Routing** | Keyword classification → route | `CLAUDE.md` `<routing>` block |
| **Skills** | Invokable procedures (plan, generate, verify, etc.) | `.claude/skills/` |
| **Agents** | Expert subagents with tool restrictions | `.claude/agents/` |
| **Hooks** | Pre/Post tool automation | `.claude/hooks/` + `settings.json` |
| **State** | Session continuity | `.sdlc/state/session.json` |

---

## How a Task Flows

```
User message
  → CLAUDE.md routing classifies it (feature / bug / refactor / ...)
  → Shows: 🎯 [TYPE] → [ROUTE] | Risk: [low|medium|high|critical]
  → Dispatches to the right pipeline:

simple  → Skill('generate') → Skill('verify') → done
medium  → Skill('plan') → wave-execute subagents → code-review → branch-finish
complex → Skill('brainstorming') → worktree → plan → wave-execute → human gate
```

## Wave Execution (Yield Pattern)

For medium/complex tasks, `Skill('subagent-driven-development')` groups tasks into waves based on `depends_on` fields in `tasks.md`:

```bash
# Automated wave computation
node .claude/scripts/wave-scheduler.js --tasks docs/changes/<feature>/tasks.md --summary
```

Tasks in the same wave are launched in **one response message** as parallel `Agent()` calls. Claude Code executes them concurrently. The orchestrator yields, awaits all results, advances `session.json revision`, and moves to the next wave.

```
Wave 1: TASK-001, TASK-003        ← no deps, run in parallel
Wave 2: TASK-002                  ← depends on TASK-001
Wave 3: TASK-004, TASK-005        ← depend on TASK-002 and TASK-003
```

## State Persistence (TaskFlow)

Session state lives in `.sdlc/state/session.json`:

```json
{
  "taskId": "feat-email-verify",
  "revision": 3,
  "lifecycle": "execute",
  "stateJson": { "currentWave": 2, "completedFiles": ["src/auth.ts"] },
  "waitJson": {},
  "activeTask": "TASK-003",
  "completedTasks": ["TASK-001", "TASK-002"],
  "decisions": []
}
```

Mutation protocol: always read `revision`, increment by 1, write atomically. If revision mismatch on write → reject (optimistic locking). On session resume: if `lifecycle = "execute"` → show recovery banner with stateJson preserved.

## Hook Pipeline

Every tool call passes through a hook chain defined in `settings.json`:

```
SessionStart  → welcome → memory-check → rtk-bootstrap → memory-decay
UserPromptSubmit → orchestrator-gate → constraint-reinforcement
PreToolUse/Agent → context-engine → provider-rotation
PreToolUse/Write|Edit → tool-policy-guard → sdd-guard → config-guard → mutation-guard
PreToolUse/Bash → safety-guard → rtk-rewrite → block-no-verify → pre-commit-guard → mutation-guard
PostToolUse/all → context-window-guard → tool-loop-detection → conversation-tick
SubagentStart → subagent-context → transcript-policy
Stop → state-sync → session-observer → rtk-analytics → context-engine
```

All hooks are non-blocking except: `sdd-guard`, `config-guard`, `safety-guard`, `block-no-verify`, `pre-commit-guard`, `tool-policy-guard`, `tool-loop-detection` (circuit breaker only).

## HANDOFF Protocol

For `/orchestrate` workflows, each agent writes a structured handoff before the next starts:

```
.sdlc/handoffs/<workflow-id>/<agent-name>-handoff.md
```

Contains: status (COMPLETE/PARTIAL/BLOCKED), findings, files modified, open questions, recommendations. No agent starts without reading the previous handoff.

---

## Context Budget Enforcement

`context-engine.py` estimates token usage before every `Agent()` call and writes:

```json
{ "percent": 35, "status": "OK", "estimatedTokens": 70000 }
```

`subagent-driven-development` reads this in Phase 0. If `percent > 40%` → do not spawn, run `/compact` first. If `percent > 80%` → hard block by the hook.

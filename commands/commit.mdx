## Overview

The `commit` skill generates commits in the Agent007 pipe-delimited format. It surveys changes, stages files by name, writes the commit, and verifies the result.

```
Skill('commit')
```

No arguments needed — `/commit` reads the current git state and guides you through the process.

---

## Commit Format

```
Tipo|IdTarea|YYYYMMDD|Descripción breve
```

| Component | Description |
|-----------|-------------|
| `Tipo` | Nature of the change (see types below) |
| `IdTarea` | Ticket / task board ID (omit if none — leave empty between pipes) |
| `YYYYMMDD` | Commit date — runs `date +%Y%m%d` automatically |
| `Descripción` | Action in English, imperative mood, max 60 characters |

### Valid types

| Type | When to use |
|------|-------------|
| `feat` | New functionality |
| `fix` | Bug correction |
| `refactor` | Restructure without behavior change |
| `review` | Code review adjustments |
| `test` | Tests added or adjusted |
| `docs` | Documentation only |
| `chore` | Maintenance, minor tasks |
| `perf` | Performance improvement |
| `ci` | CI/CD pipeline changes |
| `build` | Build system changes |
| `revert` | Revert previous commit |
| `wip` | Work-in-progress (avoid in main branches) |

## Process

### Step 1 — Survey changes

```bash
git status              # Untracked files
git diff --staged       # What's staged
git diff                # Unstaged changes
git log --oneline -5    # Recent commits (match style)
date +%Y%m%d            # Get today's date
```

### Step 2 — Stage relevant files

Prefers specific files over `git add -A` to avoid accidentally including secrets or large binaries.

Never stages: `.env`, `secrets/`, credential files, large binaries.

### Step 3 — Write commit message

```bash
git commit -m "$(cat <<'EOF'
feat|LIQUI|20260329|Implement complete CRUD for cargos module
EOF
)"
```

### Step 4 — Verify

```bash
git log --oneline -1    # Confirm commit exists
git show --stat HEAD    # Confirm correct files included
```

## Pre-Commit Guard

The `pre-commit-guard.py` hook (registered in `settings.json`) intercepts every `git commit` and blocks:

- Staged `.env` files
- Hardcoded secrets (`API_KEY=`, `SECRET=`, `PASSWORD=`, `TOKEN=` with literal values)
- JS `debug` statements (`debugger` keyword)
- Commits that don't follow the pipe-delimited format

The `block-no-verify.py` hook blocks all `git commit --no-verify` calls — bypassing the guard is not allowed.

---

## Pre-Commit Checklist

- Format: `Tipo|IdTarea|YYYYMMDD|Descripción`
- Description is clear and concise (max 60 characters)
- Code is tested
- No linting errors
- No secrets or credentials in diff

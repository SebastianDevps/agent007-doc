## Overview

The `pull-request` skill creates a GitHub pull request with a structured body: linked issue, summary bullets, motivation, test plan, breaking changes, and rollback plan. **Issue-first**: every PR must link an approved issue before it can be created.

```
Skill('pull-request')
```

Invoked after `finishing-a-development-branch` selects "Push + PR", or directly when you want to open a PR.

---

## Step 0 — Issue-first gate

Every PR MUST link an approved GitHub issue.

```bash
gh issue view <N> --json title,labels -q '{title: .title, labels: [.labels[].name]}'
```

The skill STOPS if any of these fail:

- Issue exists and is not closed
- Issue has the `status:approved` label (maintainer-only — wait if missing)
- Issue number known — will add `Closes #<N>` in the PR body

If no issue exists, run `Skill('issue-creation')` first.

## Process

### Step 1 — Survey branch

```bash
git status
git log main..HEAD --oneline     # All commits in this branch
git diff main...HEAD --stat       # Files changed vs main
```

### Step 2 — Verify tests pass

```bash
npm test  # or pytest / go test / cargo test
# Must see 0 failures before creating PR
```

### Step 3 — Push branch

```bash
git push -u origin $(git branch --show-current)
```

### Step 4 — Create PR

Title format: `tipo — Descripción imperativa en inglés (max 70 chars)`

Examples:
```
feat — Add user authentication with JWT refresh rotation
fix — Resolve null pointer in UserService.findById
refactor — Extract payment service to shared module
```

```bash
gh pr create --title "tipo — Descripción imperativa" --body "$(cat <<'EOF'
## Linked Issue

Closes #<N>

## Summary

- [Bullet 1: what was implemented and why]
- [Bullet 2: key decision made]
- [Bullet 3: what was changed / removed]

## Motivation

[Why this change was needed — link to issue/ticket]

## Test Plan

- [ ] [Test scenario 1]
- [ ] [Test scenario 2]
- [ ] [Edge case tested]
- [ ] Existing tests pass (`npm test`)

## Breaking Changes

[None / Describe if any]

## Rollback Plan

[How to revert if this breaks production — e.g. `git revert <sha>`, feature flag off]
EOF
)"
```

<Warning>
  The PR body must NEVER contain `Co-Authored-By:` footers, tool attribution lines (`Generated with...`, `🤖 ...`), or external branding. The pre-commit and pre-PR guards reject these.
</Warning>

### Step 5 — Confirm

```bash
gh pr view  # Show PR URL and status
```

---

## Checklist before creating

- Linked issue with `status:approved`
- PR body contains `Closes #<N>`
- Branch pushed to remote
- Tests passing (0 failures)
- Title ≤ 70 chars, `tipo — Descripción` format
- Summary bullets are concrete (no "changed some files")
- Test plan is concrete (no "tested manually")
- No secrets or credentials in diff
- No `Co-Authored-By` / tool attribution / external branding

---

## Hard rules

- NEVER `Co-Authored-By` footer
- NEVER tool attribution lines (`Generated with...`, `🤖 ...`)
- NEVER ticket IDs, tool names, or branding in the title
- NEVER force push to `main` / `develop`
